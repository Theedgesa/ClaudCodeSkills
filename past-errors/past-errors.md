# Past Errors — Lessons Learned

Core lessons from PR #19-25 cascade (7 reactive PRs for one fix).

---

## Key Rules

1. **One comprehensive PR > multiple hotfixes** — Analyze full codebase impact before coding
2. **Never create profiles for guest customers** — profiles.id FK → auth.users. See option1-auth.md
3. **Align code to database, not vice versa** — DB is source of truth
4. **Check all constraints before INSERT** — CHECK, NOT NULL, UNIQUE, FK
5. **Provide explicit UUIDs for auth tables** — Never assume DEFAULT gen_random_uuid()
6. **Consistent indexing convention** — Pick 0-based or 1-based, never mix
7. **Supabase executes first statement only** — Always BEGIN/COMMIT. See supabase-patterns.md
8. **Guard env vars in migration scripts** — CLIENT_URL=localhost breaks email links. See migration-script-urls.md
9. **Verify after EACH phase** — Don't batch verification to the end
10. **Test locally before PR** — Production is not a test environment
11. **Restart server before verifying new routes** — New backend routes/services don't exist until the running server is restarted. Always kill + restart the dev server before testing API endpoints.
12. **Verify data retrieval end-to-end, not just DB** — After DB migrations, verify data is accessible through the full stack: DB → service → controller → API endpoint (with auth token) → confirm JSON response has expected rows. Never assume "table has data" means "frontend can load it".
13. **Run per-phase Self-UAT before proceeding** — Every plan phase must include executable Self-UAT tests. Do not proceed to the next phase until ALL tests pass. If any test fails, fix the issue and re-run ALL tests for that phase.
14. **Always use structured logging in new code** — Never raw `console.log` in new feature code. Create feature-scoped loggers (see `server/utils/zohoLogger.js` pattern) that output JSON-structured log entries with timestamps, levels, component tags, and event names. Existing `console.log` calls in untouched code are left alone (separate concern).
15. **Verify column constraints before changing INSERT values to NULL** — When code changes INSERT values to NULL, verify `is_nullable` via `SELECT is_nullable FROM information_schema.columns WHERE table_name = 'X' AND column_name = 'Y'` before implementation. Include migration if column is NOT NULL. Never trust code comments about DB state — query `information_schema`.
16. **Trace data through every transformation step** — When data passes through destructure → rebuild (e.g., `req.body → processedItems.push → loop`), document the full variable chain: `source → transform → destination` with line numbers. At each arrow, verify the property exists on the receiving object. A missing property in a `push()` call is silent — no error, just `undefined`.
17. **Test one order per payment method after order flow changes** — After any change to order processing, verify with one test order per payment method: Storefront MyFatoorah, Reception Manual POS, Reception MyFatoorah, Reception Auto POS (code review if not live). A passing test on one workflow does NOT validate others — each has a different entry point and data transformation chain.
18. **Never assume or guess why something behaves a certain way** — Always check logs, trace execution, and prove root cause with evidence. Never speculate to the user.
19. **Never hallucinate answers** — If you don't know something, say so. Never fabricate test results, file contents, or explanations.
20. **Test through the frontend, not curl** — All Self-UAT must simulate real user actions through the actual frontend pages (localhost:3000/3001), not via curl/API calls. Curl tests miss broken buttons, modals in wrong conditionals, missing toasts, stale caches, and state not refreshing. API/DB queries are supplementary verification only. Do NOT restart servers between tests.
21. **Never bypass errors — study every one** — When encountering ANY error during implementation (pre-existing or newly introduced), STOP and: (a) read the full error message + stack trace, (b) identify the root cause, (c) assess impact on the current workflow/plan being executed, (d) if it impacts the current work → add to plan and fix before proceeding, (e) if no impact on current work → add to backlog with severity + context. Never dismiss errors as "pre-existing" or "not related" without evidence. An error you ignore today becomes the production incident you debug tomorrow.
22. **Critique plans against code AND documentation, not just against themselves** — Before writing or approving ANY plan that touches existing code or external APIs: (a) READ the actual source files that will be modified — use `sg` and `search_graph` to find hardcoded values, `process.env` usage, algorithm assumptions, header names, and API format expectations, (b) READ the official external API documentation for every integration mentioned (MyFatoorah, Zoho, Tamara, Shopify, etc.) — verify header names, signing algorithms, webhook formats, request/response structures against our implementation, (c) LOAD the relevant source code AND documentation into context before making any claims about how the system works. Never assume existing code is correct — verify it. A plan that says "webhook signature verification works" can hide a fundamentally wrong implementation (wrong header name, wrong HMAC algorithm, wrong body format). The PROJ-056 critique loop ran 6 agents across 2 rounds and caught 37 plan-level gaps but missed every implementation-level blocker (wrong MF header name, wrong V2 signing algorithm, missing success page verify, env-var dependency in webhook controller) because no agent read the actual code or MF's documentation.

    This rule extends beyond files you modify to the FULL BLAST RADIUS of your change:
    (a) Functions you CALL — verify method names exist in the actual service file, not architecture docs. `rbac.md` documented `hasPermission()` for 2 months while the actual method was always `userHasPermission()`. Architecture docs drift.
    (b) Functions that CALL YOU — use `trace_call_path` or `sg -p '$_.functionName($$$)' server/ --lang js` to find existing callers of any function you modify. Verify your changes don't break their assumptions about return shapes or side effects.
    (c) Functions ADJACENT in the workflow — read the execution chain before and after your code. Verify middleware sets what you expect on `req`, verify the data shape at your call site.
    (d) Response shapes being CONSUMED — if adding a field to a backend response, verify the frontend type includes it AND any hardcoded response mapping passes it through (the searchMembers bug: column was fetched then dropped by a hardcoded `.map()` that didn't include the new field).

    Incident log: PROJ-056 (wrong webhook header — doc said X, code said Y), PROJ-091 searchMembers (SELECT had field, response mapping dropped it), PROJ-091 hasPermission (doc said `hasPermission`, code was `userHasPermission` since day one).
24. **Self-UAT gates are non-negotiable — no deploy without evidence** — Before ANY deployment (`scp`, `rsync`, `pm2 restart`, `git push`), EVERY Self-UAT test from the current phase MUST have been executed with actual output documented. "node --check passes" is NOT a substitute for integration tests. A pure-function test is NOT a substitute for API-level or browser-level tests. If a Self-UAT says "curl 10 times → 11th returns 429", you MUST actually curl 11 times and paste the output. If it says "dashboard shows Active", you MUST load the dashboard (locally or via API) and verify. Skipping a Self-UAT because "it probably works" is the same failure mode that caused PROJ-056's webhook header bug — assuming correctness instead of proving it. The implement skill's "Gate: All Self-UAT must pass before proceeding" is a HARD GATE, not a suggestion.

25. **Never instantiate SDK clients at module scope in Next.js pages** — `createClient()`, `new Stripe()`, or any SDK that throws when config is missing must be created inside a function or component, never at the top level of a file. Next.js static export (`output: "export"`) evaluates module-scope code at build time. If `NEXT_PUBLIC_*` env vars are absent during build (CI, clean worktree), the SDK throws and kills the entire build. Use a lazy singleton: `let _client = null; function getClient() { if (!_client) _client = createClient(...); return _client; }`. Incident: PROJ-095 — `auth/confirm` and `reset-password` pages had `const supabase = createClient(process.env.NEXT_PUBLIC_SUPABASE_URL || '', ...)` at module scope. Build crashed in any environment without `.env` files. Went undetected because (a) EC2 builds have env vars, (b) `npm run dev` skips SSG, (c) CI had no `npm run build` step.
26. **Any change to dependency installation must update CI workflow** — When modifying `package.json` workspaces, adding/removing packages, changing install flags, or restructuring `node_modules` resolution: (a) READ `.github/workflows/quality.yml` and verify every `Install dependencies` step still works with the new config, (b) check for hardcoded `npm ci --legacy-peer-deps`, `cd X && npm ci`, or separate install steps that may now be redundant or broken, (c) verify that CI's install path matches what you tested locally — if you ran `npm install` separately in a subdirectory locally, CI probably doesn't do that. **Local ≠ CI:** Local dev often has pre-populated `node_modules` from previous installs. CI starts clean every time. A quality gate passing locally but failing in CI means the local environment masked a real dependency resolution gap. Incident: PROJ-155 — added `reception-website` to workspaces, local quality passed because `store-website/node_modules` was populated from a prior `npm install`. CI failed because the workflow still had a stale `npm ci --legacy-peer-deps` step for reception and never installed store-website deps properly.
23. **Always audit ALL env files before any frontend deployment** — When a plan touches frontend code that makes API calls, ALWAYS read and compare ALL `.env*` files: `.env`, `.env.local`, `.env.production`, `.env.development`, `.env.production.local`. Next.js env priority: `.env.local` OVERRIDES `.env.production` even in production builds. A missing `NEXT_PUBLIC_API_URL` in `.env.production` caused the entire storefront to call `localhost:5001` in production — every frontend API call silently failed, dashboard showed broken profile modal, success page verify never fired. The payment flow only worked because webhooks are server-to-server and don't use the frontend URL. **Before every frontend build for deployment:** (a) `cat .env.production` and verify ALL `NEXT_PUBLIC_*` vars are set to production values, (b) `grep -r 'localhost' out/_next/static/` AFTER build to catch any baked-in dev URLs, (c) if building locally for production deploy, use explicit env override: `NEXT_PUBLIC_API_URL=https://api.example.app/api npm run build`. Add env file verification to the Agent 0 checklist for any plan that deploys frontend code.

## Self-UAT Enforcement (Mandatory)

**Before any deploy command (`scp`, `rsync`, `pm2 restart`, `git push`):**
1. Each phase's Self-UAT table must have EVERY row verified with actual command output
2. Playwright test files from the plan must be CREATED and RUN before deploy
3. "Syntax passes" counts as 1 test — it does NOT substitute for the other N-1 tests
4. If a test requires a running server, start the server and run the test — don't mark it "PENDING"
5. If you catch yourself typing `scp` or `pm2 restart` without having run all Self-UATs, STOP

**PROJ-057 incident:** All 4 phases implemented and deployed to production with 13 Self-UAT tests skipped and the entire Playwright phase not even created. Pure-function test was treated as sufficient for the whole phase. Root cause: no enforcement hook + context fatigue + false confidence from partial testing.

## Verification Checkpoints (Mandatory)

**After DB migration:** Run SELECT to verify, test FK constraints
**After backend changes:** RESTART dev server, then verify server starts without crash, `sg -p 'removedPattern' server/ --lang js` for removed patterns → zero results
**After new routes/controllers:** Restart server, hit each new endpoint with auth token, verify JSON response contains expected data (not just 200 status — check row counts and field values)
**After frontend types:** `npx tsc --noEmit` → zero errors
**After frontend pages:** `npm run build` after every 2-3 page changes
**After workspace/dependency changes:** Read `.github/workflows/quality.yml` and verify all install steps are compatible. Check for stale `--legacy-peer-deps`, hardcoded `cd X && npm ci`, or missing workspace members.
**Final:** Both frontends build, end-to-end verification with running server — load the actual page in browser or curl all new endpoints with auth

34. **Production-to-staging database mirrors must use `--data-only`** — When mirroring production data to staging via `pg_dump`, ALWAYS use `--data-only` flag for application schemas (public, contact_service, finance). A full dump (without `--data-only`) includes DDL: CREATE TABLE, ALTER TABLE ADD CONSTRAINT, CREATE TRIGGER, CREATE FUNCTION. This restores production schema over staging schema changes (FK repointing, column alterations, trigger drops). The auth schema should also use `--data-only` to preserve Supabase's GoTrue structure. After ANY mirror restore, verify: (a) FK constraints still point to profiles (not auth.users), (b) `profiles.tenant_id` is NOT NULL, (c) `handle_new_user` trigger is absent, (d) `payment_gateway_config.mode` is `sandbox`. If schema was accidentally restored, re-run the FK migration from `.claude/work/PROJ-154-staging-fk-repointing/production-migration.sql`. Incident: PROJ-154 — June 7 mirror used `pg_dump -n public -n contact_service -n finance -n supertokens` without `--data-only`, restoring all 20 production FK constraints over PROJ-148's repointing. Discovered June 8 during payment testing (23503 FK violation on order_items INSERT).

35. **Safety net / catch-up queries MUST have temporal lower bounds** — Any cron, setInterval, or scheduled function that queries for "missing", "unprocessed", or "orphaned" records MUST include a `created_at > startTime` lower bound. Without it, the first startup against a database with historical data will process the ENTIRE backlog. Acceptable lower bounds: process start time (`new Date().toISOString()` captured at boot), deployment date, or last-processed timestamp from DB. Also require `.limit(N)` to prevent event loop blocking on large result sets. Incident: PROJ-143 safety net cron queried all paid orders >2 min old with no lower bound. On first boot against staging (2,056 paid orders), it sent 125 order confirmation emails to historical customers in 11 minutes. The plan pseudocode had the same bug.

36. **Route rewrites must preserve the middleware chain** — When rewriting or replacing an Express route handler, ALWAYS verify the middleware chain from the original route (`isAuthenticated`, `requirePermission`, `attachLocationScope`, etc.) is carried over to the new handler. Read the existing route definition BEFORE rewriting. If the plan's security section (§14) states middleware requirements, the implementation section (§15) pseudocode MUST show them in the route definition. Add a UAT test that verifies middleware on any modified route: `sg -p 'router.$METHOD($PATH, isAuthenticated, $$$)' server/routes/<route-file> --lang js`. Incident: PROJ-143 rewrote `/sync-order` and `/sync-orders` endpoints without `isAuthenticated` despite plan §14 requiring it and the original routes having it.

37. **Plan pseudocode `object.property` must be verified against actual class API** — When a plan references `object.property.method()`, use `sg -p 'this.propertyName' path/to/class.js --lang js` to verify the property exists on `this` (not as a local variable inside some method). Architecture docs and plan pseudocode drift from implementation. If `property` is a local `const/let` inside a method body, it cannot be accessed externally — the plan must use a different approach (create a new instance, expose the property, or restructure). Incident: PROJ-143 plan referenced `provider.svc.getInvoicePDF()` but `ZohoProvider.createInvoice()` creates `const svc = new ZohoBooksService(...)` as a local variable never assigned to `this.svc`. Result: `Cannot read properties of undefined` at runtime.

39. **Never call a method that doesn't exist on a required module — and never mask it with try/catch** — Before writing `service.method()`, run `sg -p 'async method($$$) { $$$ }' <service-file> --lang js` to verify the method is defined in the module. If a method doesn't exist, the call will throw `TypeError: service.method is not a function`. A try/catch with a hardcoded fallback converts this crash into a silent success — the code runs, tests pass, the fallback produces correct results, and the dead primary path survives indefinitely. The `require-check` quality gate (phase 2) now detects this statically across `server/` and `finance-service/`. Incident: PROJ-082 (May 7, 2026) called `gatewayConfigService.getGatewayAccount()` — method never existed on `GatewayConfigService`. The try/catch fell back to hardcoded Zoho account IDs. Survived 33 days, 6 PRs (including PROJ-143 which modified the same file), and deployed to production EU-Region on June 9. 159 warnings in the first hour. The plan process was active since March 19 but PROJ-082 had no plan directory — enforcement gap. Detected by extending `require-check.mjs` to scan `finance-service/` and filtering by resolved path instead of relative path string.

43. **Verify JSONB payload keys match consumer destructure patterns before deploying DB triggers** — When a DB trigger builds a JSONB payload (`jsonb_build_object`), use `sg -p 'const { $$$FIELDS } = $EXPR' path/to/consumer.js --lang js` to find how the consumer destructures the payload. The trigger SQL must use the same key casing. A snake_case key in the trigger and a camelCase destructure in the consumer will silently produce `undefined` — no type error, no runtime crash, just missing data. Incident: PROJ-161 — trigger used `'order_id'` (snake_case), consumer used `const { orderId } = jobPayload` (camelCase). All Zoho invoices blocked for 24h, 9 dead jobs. Fix: 1 SQL `CREATE OR REPLACE FUNCTION` to change the key.

45. **Retry/safety-net mechanisms MUST have three guards** — (a) Max retry count per item (cap at 3-5). (b) Circuit breaker: if N consecutive attempts fail with the same error class (e.g., SMTP 550), pause the entire mechanism for a cooldown period. (c) Dedup must match BOTH success AND failure records — if dedup uses field X, verify that X has the same format in both the success log path and the failure log path. The PROJ-143 safety net had none of these: `.catch(() => {})` swallowed errors, no retry counter, and dedup used `ilike('subject', ...)` which only matched success logs (rendered subject) not failure logs (raw template slug). Result: 5,899 failed retry attempts over 2.5 days, Zoho SMTP blocked for the entire organization. Incident: PROJ-168.

51. **Always verify database schema state via `information_schema` BEFORE writing or executing migration plans** — A plan that says "column X does not exist" may be based on stale PM2 error logs from before a previous migration. Run `SELECT column_name FROM information_schema.columns WHERE table_name = 'X' AND column_name = 'Y'` and adapt the plan if the column already exists. If the column exists but errors still occur, the issue is PostgREST schema cache staleness (see debugging pattern #9), not a missing column. Hook: `pre-migration-verify.sh` fires on `apply_migration` to remind. Incident: PROJ-170 — plan assumed 4 email tables lacked tenant_id columns. `proj_042_add_tenant_id` had added them 3 months earlier (March 29). Migration was adapted to only update constraints.

52. **Always `grep -rn '<<<<<<'` after git rebase before committing** — Rebase can partially auto-resolve a hunk and leave a dangling `<<<<<<< HEAD` marker with no matching `=======` or `>>>>>>>`. The file passes `node --check` (it's valid JS if the marker is inside a string or comment). Hook: `post-rebase-conflict-guard.sh` blocks `git commit` and `git rebase --continue` if staged files contain conflict markers. Incident: PROJ-170 — leftover `<<<<<<< HEAD` in `email.service.js` after rebase, committed and pushed before caught.

53. **Always verify PM2 `created_at` timestamps are AFTER the last `git pull` on deploy** — `git pull` brings new code to disk but PM2 processes keep running old code in memory. After every `git pull`, restart ALL affected PM2 processes (`pm2 restart main-api finance-service`) and verify: `pm2 describe <name> | grep created` must show a timestamp after the pull. A process with 34h uptime after a 24h-old pull means it never restarted. Hook: `pm2-restart-guard.sh` reminds on SSH git pull commands and warns if only one service is restarted. Incident: PROJ-170 — `finance-service` was not restarted after production `git pull`. The `verification` event path kept failing with `tenant_id NOT NULL` for 34 hours until caught by `/evidence` check.

54. **Use MCP browser tools for interactive UI testing, not Playwright scripts** — When testing a UI flow for the first time on a live environment, use `browser_navigate` / `browser_snapshot` / `browser_click` (MCP Playwright tools) for step-by-step exploration. Playwright scripts require knowing element selectors upfront — writing scripts blind causes iterative failures (wrong selectors, React state issues, overlay interceptions, iframe nesting). MCP browser tools give you the accessibility snapshot with exact element refs before each click. Use PW scripts only for repeatable regression suites after the flow is proven interactively. Hook: `playwright-script-guard.sh` blocks `npx playwright test` and `.spec.ts` file creation. Incident: PROJ-170 — 6 failed PW script iterations (3+ hours) vs 1 successful MCP browser session (15 minutes) for the same storefront payment flow.

55. **Verify tenantId threading through ALL internal method calls, not just controller→service** — When `.eq('tenant_id', tenantId)` is added to a query, trace EVERY caller up to the entry point. The `quality:tenant-caller` gate only checks controller→service calls. It does NOT catch: (a) service→service internal calls where a method calls another method without forwarding tenantId, (b) module-level cached variables used instead of parameters (`_cachedTenantId`), (c) controller methods that spread `req.body` without injecting `getTenantId(req)`. The 5-minute cache and pass-only order dominance mask these bugs — they only surface on cold starts with non-pass items (rentals, classes, products). Incident: PROJ-171 session found 3 instances in one day: (1) `_cachedTenantId` in webhook.controller.js only set inside processPassPurchase, (2) `createClassSchedule`/`updateClassSchedule` passing raw req.body without tenantId, (3) finance-service `location.service.js` internal methods calling `getActiveLocations()` without tenantId. All 3 were added by PROJ-MT5 migration and survived months because the cache or item-type distribution hid them.

56. **Always destructure `{ error }` from Supabase `.update()` and check it** — PostgREST silently strips unknown columns from PATCH requests. If `.update({ known_col: val, unknown_col: val })` is sent, PostgREST drops `unknown_col` without error — returns 204 success, but only `known_col` is written. The code sees no error and returns 200 to the user. This creates a silent data loss bug that passes all testing (the endpoint "works"). Always: `const { error } = await supabaseAdmin.from('table').update({...})` and `if (error) { log + return 500 }`. Incident: PROJ-179 — `used_token_jti` column didn't exist, `.update({ verified_email: true, used_token_jti: jti })` silently dropped both fields for 17 days. 50+ users saw "Email verified successfully" while DB stayed `false`. Zero errors in logs.

57. **After adding a DB column, verify it's writable via PostgREST before deploying code that uses it** — PostgREST caches the table schema. After `ALTER TABLE ADD COLUMN`, the column exists in `information_schema` but PostgREST may not know about it yet. Test: `supabaseAdmin.from('table').update({ new_col: 'test' }).eq('id', testId)` then `.select('new_col')` — if the value comes back `undefined` or `null` when you wrote a non-null value, the cache hasn't refreshed. On managed Supabase, `NOTIFY pgrst, 'reload schema'` doesn't always propagate through the connection pooler. Always test on the target environment (production server), not locally — production and local may have different cache states. Incident: PROJ-179 — column added via `execute_sql`, local PostgREST couldn't write to it for 30+ minutes. Production PostgREST had already refreshed and worked immediately.
